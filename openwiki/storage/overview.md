# Storage & Data Models Overview

CareerOps uses three data stores, each optimized for its use case:

| Store | Technology | Purpose | Scope |
|-------|-----------|---------|-------|
| **Applications & Config** | PostgreSQL (Prisma) | Applications, users, portals, keywords, sessions | Multi-user |
| **Evaluation Reports** | MinIO (S3-compatible) | Full A–F evaluation reports (markdown) | Per-application |
| **Candidate Context** | DynamoDB | CV and profile data (JSON) | Per-user |

## PostgreSQL (Prisma)

Postgres is the **primary persistent store** for the application. It uses Prisma ORM for type-safe queries and migrations.

### Core Tables

#### User
Multi-user support via NextAuth v5. Stores account info and resume/profile references.

```prisma
model User {
  id            String    @id @default(cuid())
  email         String?   @unique
  name          String?
  image         String?
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
  
  resumeKey       String?         // MinIO S3 key: "resumes/<id>.pdf"
  resumeUpdatedAt DateTime?
  
  accounts       Account[]
  sessions       Session[]
  applications   Application[]    // User owns these apps
  filterKeywords FilterKeyword[]  // User's title-filter keywords
}
```

#### Application
The core job application record. Each row represents a discovered or applied-for job.

```prisma
model Application {
  id        String    @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  userId    String    @index
  date      String    // "2024-01-15"
  company   String    // "Anthropic"
  role      String    // "Senior Platform Engineer"
  score     String    // "4.5/5" (stored as string for display)
  status    AppStatus // Enum: Evaluated, Applied, Interview, Offer, etc.
  pdf       String    // "✅" or empty (indicate CV generated)
  reportName    String  // MinIO object name: "001-anthropic-2024-01-15.md"
  reportUrl String?   // Public MinIO URL or presigned URL
  url       String?   @unique(map: "unique_user_url") // Job posting URL
  createdAt DateTime  @default(now())
  updatedAt DateTime  @updatedAt
  
  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@unique([userId, url])  // Unique by user + job URL
  @@index([userId, createdAt desc])
}
```

**Key Points:**
- **Unique constraint:** `(userId, url)` prevents duplicate applications for the same user
- **Score stored as string:** "4.5/5" allows display formatting without calculation
- **Report reference:** `reportName` stores MinIO object key, `reportUrl` is the public URL
- **Status enum:** Type-safe status values (generated by Prisma from the enum)

#### Portal
Scan targets (company career pages). Global, shared across all users.

```prisma
model Portal {
  id         String   @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  name       String   @unique  // "Anthropic", "Google", etc.
  careersUrl String?           // https://jobs.anthropic.com/...
  api        String?           // Custom API endpoint (Greenhouse, etc.)
  enabled    Boolean  @default(true)
  createdAt  DateTime @default(now())
  updatedAt  DateTime @updatedAt
}
```

**Usage:**
- `npm run portals -- list` queries this table
- `npm run scan` reads all enabled Portals
- Each Portal maps to one company/job board

#### FilterKeyword
Title-filter keywords. Scoped per user (allow different users to have different filters).

```prisma
model FilterKeyword {
  id     String @id @default(dbgenerated("gen_random_uuid()")) @db.Uuid
  userId String @index
  kind   String // "positive" or "negative"
  value  String // "software engineer", "sales engineer", etc.
  
  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@unique([userId, kind, value])
}
```

**Usage:**
- `npm run portals -- keywords list` queries this table
- `npm run scan` loads keywords and uses them to filter jobs
- Positive keywords are required (at least one); negative keywords are optional

#### Account, Session (Auth.js)
NextAuth v5 tables for OAuth session management. Auto-managed by NextAuth.

```prisma
model Account {
  id                String  @id @default(cuid())
  userId            String
  type              String
  provider          String
  providerAccountId String
  refresh_token     String? @db.Text
  access_token      String? @db.Text
  expires_at        Int?
  token_type        String?
  scope             String?
  
  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
  @@unique([provider, providerAccountId])
}

model Session {
  id           String   @id @default(cuid())
  sessionToken String   @unique
  userId       String
  expires      DateTime
  
  user User @relation(fields: [userId], references: [id], onDelete: Cascade)
}
```

### Indexes

Key indexes for performance:

```
Application:
  - (userId, url) UNIQUE — Deduplication
  - (userId, createdAt DESC) — List/sort by user

FilterKeyword:
  - (userId, kind, value) UNIQUE — Dedup keywords per user
  - userId — Find keywords for a user
```

### Migrations

Manage schema changes with Prisma:

```bash
# Create a new migration
npx prisma migrate dev --name <descriptive-name>

# Apply migrations to production
npx prisma migrate deploy

# Check current schema
npx prisma db push
```

## MinIO (S3-Compatible Object Storage)

MinIO stores evaluation reports as markdown files. It's S3-compatible, so any S3-like service works (AWS S3, DigitalOcean Spaces, etc.).

### Report Storage

**Naming convention:**
```
{###}-{company-slug}-{YYYY-MM-DD}.md
```

**Examples:**
```
001-anthropic-2024-01-15.md
002-google-2024-01-16.md
003-openai-2024-01-17.md
```

**Location:** Bucket defined by `MINIO_BUCKET` env var (default: `careerops`)

### Report Access

Reports are stored in MinIO and accessed via:

1. **Internal reference:** `Application.reportName` (MinIO object key)
2. **Public URL:** `Application.reportUrl` (http or https URL)

```
Application.reportName = "001-anthropic-2024-01-15.md"
Application.reportUrl  = "http://localhost:9000/careerops/001-anthropic-2024-01-15.md"
```

The URL allows:
- Direct browser access (if report is public)
- Embedding in web dashboard
- Sharing with others

### Configuration

```bash
# .env
MINIO_ENDPOINT=http://localhost:9000        # MinIO API endpoint
MINIO_ACCESS_KEY=admin
MINIO_SECRET_KEY=careerops123
MINIO_BUCKET=careerops                      # Bucket name
MINIO_PUBLIC_ENDPOINT=http://localhost:9000 # Optional: browser-reachable URL
```

### Usage in Code

```typescript
import { uploadReport, downloadReport, getReportUrl } from "@/lib/minio";

// Upload a report
const key = await uploadReport({
  reportName: "001-acme-2024-01-15.md",
  markdown: "# Evaluation...",
});

// Download a report
const markdown = await downloadReport("001-acme-2024-01-15.md");

// Get a public URL
const url = getReportUrl("001-acme-2024-01-15.md");
```

## DynamoDB (Optional, Candidate Context)

DynamoDB stores CV and profile data (optional). Falls back to local files if not configured.

### Tables

#### CVs Table

Stores candidate CV as JSON.

```
Partition Key: userId (String)
Attributes:
  cv (String): Full CV markdown
  uploadedAt (Number): Unix timestamp
```

**Usage:**
```typescript
import { getCV, putCV } from "@/lib/cv-store";

// Load CV
const cv = await getCV(userId);

// Update CV
await putCV(userId, cvMarkdown);
```

#### Profiles Table

Stores candidate profile (target roles, compensation, location, etc.) as JSON.

```
Partition Key: userId (String)
Attributes:
  profile (String): YAML or JSON profile data
  updatedAt (Number): Unix timestamp
```

**Usage:**
```typescript
import { getProfile, putProfile } from "@/lib/profile-store";

// Load profile
const profile = await getProfile(userId);

// Update profile
await putProfile(userId, profileYaml);
```

### Configuration

```bash
# .env
DYNAMODB_ENDPOINT=http://localhost:8000    # DynamoDB Local (dev)
DYNAMODB_REGION=us-east-1
DYNAMODB_TABLE_CV=CVs                      # Table name
DYNAMODB_TABLE_PROFILE=Profiles            # Table name
AWS_ACCESS_KEY_ID=local                    # Dummy for local dev
AWS_SECRET_ACCESS_KEY=local                # Dummy for local dev
```

### Local Development

DynamoDB Local is included in `docker-compose.yml`:

```bash
docker compose up dynamodb
npm run dynamo:init  # Initialize tables
npm run dynamo:cv    # Upload CV to DynamoDB
npm run dynamo:profile  # Upload profile to DynamoDB
```

## Data Flow Summary

```
Scan Pipeline:
  Postgres (Portal, FilterKeyword)
    ↓
  Scanner finds jobs
    ↓
  Postgres (Application, status=N/A)

Evaluate Pipeline:
  Postgres (Application, N/A)
    ↓
  DynamoDB (CV, Profile)
    ↓
  LLM evaluation
    ↓
  MinIO (store report)
    ↓
  Postgres (Application, score + reportUrl updated)

Web Dashboard:
  Postgres (Application) — list/update
    ↓
  MinIO (reports) — view
    ↓
  DynamoDB (CV, Profile) — optional enrichment
```

## Migration & Backup

### Backup Postgres

```bash
# Dump database
pg_dump -h localhost -U careerops careerops > backup.sql

# Restore from dump
psql -h localhost -U careerops careerops < backup.sql
```

### Backup MinIO

```bash
# Download all reports
mc cp --recursive minio/careerops ./backup-reports/
```

### Backup DynamoDB

```bash
# Export table (boto3 / AWS SDK)
aws dynamodb scan --table-name CVs > cvs-backup.json
aws dynamodb scan --table-name Profiles > profiles-backup.json
```
