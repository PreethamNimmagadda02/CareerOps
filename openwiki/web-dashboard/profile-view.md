# Profile View – User Interface Documentation

This document describes the **Profile View** page (`web/components/profile-view.tsx`) in the CareerOps web dashboard. It explains the UI layout, the sections presented to the user, the data model they edit, and the interaction flow.

---

## Overview

- **Entry point**: `/profile/page.tsx` renders `ProfileView`.
- **Purpose**: Let the user manage personal data, career information, work preferences, job‑matching settings, and CV details.
- **Key UI features**:
  - Profile‑completeness banner with a progress bar.
  - Sticky quick‑section navigation bar.
  - Inline edit mode per section with **Save** / **Cancel** actions.
  - Automatic résumé upload → extraction → populate fields.
  - Deterministic matching defaults via `buildMatchingPrefs` (see `web/lib/matching-defaults.ts`).
  - Server‑side validation via `src/lib/profile-validation.ts` (still the source of truth for scan eligibility).

### Profile Completeness Banner

- Computed from `buildChecklist(profile, cv, resumeKey)`.
- Shows **completed / total** items and a percentage bar.
- Clicking a missing item scrolls to that section.

### Quick Section Navigation

- Sticky bar (`<nav>`) lists all section anchors (`SECTION_NAV`).
- Highlights the section currently in the viewport via an `IntersectionObserver`.
- Clicking a button scrolls smoothly to the target.

### Auto‑Fill from Résumé

1. User uploads a résumé file.
2. Backend extracts profile + CV data (`/api/profile/resume/extract`).
3. `onExtracted` updates UI state, populating fields such as:
   - `headline`
   - `professional summary` (`exit_story` ↔ `cv.summary`)
   - Target roles, location, etc.
4. Users can re‑run the extractor at any time via the **Auto‑fill** button.

### Matching Defaults (`buildMatchingPrefs`)

- Consolidates user‑entered titles, avoid list, preferred locations, and remote flag into the `matching` object.
- Handles legacy fields (`exclude_titles`, `seniority_exclusions`) by merging them.
- Guarantees deterministic output used by the scanner.

### Removed / Consolidated Fields

- **Twitter** field removed from `candidate` (no longer displayed).
- **Archetype `level`** and **fit** UI collapsed; only `name` and a simplified `fit` selection remain.
- Duplicate location fields merged – city/country lives solely in *Personal Info*; onsite availability merged into a free‑form *flexibility* field.

## Data Flow

- **Load**: `GET /api/profile` → populates `user`, `profile`, and `cv` state.
- **Save**: `PATCH /api/profile` with a JSON body containing the edited slice, e.g. `{ profile: { candidate: { … } } }` or `{ cv: { experience: [...] } }`.
- **Resume upload**: `POST /api/profile/resume` (multipart) → returns `resumeKey` and timestamp.
- **Resume delete**: `DELETE /api/profile/resume`.

All saves trigger a toast notification (`useToast`) and refresh the in‑memory state.

## Extending / Maintaining

- To add a new field, extend the corresponding `ProfileData` / `CvData` type and update:
  1. Draft state variables.
  2. `start*` helper to seed drafts.
  3. UI component (e.g., `Field` or `ChipsField`).
  4. `save` call payload.
- Keep UI changes in sync with backend validation (`src/lib/profile-validation.ts`).
- Update this OpenWiki page to reflect any UI/field changes.
